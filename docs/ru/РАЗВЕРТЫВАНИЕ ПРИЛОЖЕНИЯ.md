# Руководство по развёртыванию hiTech

## Область применения
- Описывает развёртывание платформы доставки дронами (оркестратор на Go, сервис дронов на Python, админ-панель, хранилище, система обмена сообщениями и стек наблюдаемости).
- Применимо для локальной разработки, стенда и продакшена с использованием Docker Compose.
- Предполагает Linux-хосты с Docker Engine ≥ 24 и плагином Docker Compose ≥ 2.20.

## Предварительные требования
- Установлены и запущены Docker Engine и Docker Compose CLI.
- Доступ по Git для клонирования репозитория `hiTech`.
- Учетные данные для:
  - PostgreSQL (первичный superuser и целевая БД);
  - MinIO (root user/password);
  - RabbitMQ (admin user/password);
  - SMS Aero API (email и ключ) для отправки SMS;
  - Firebase service account (JSON-файл), если нужны push-уведомления.
- DNS-записи (или записи в `hosts`) для публичных сервисов при работе в проде с Nginx.

## Структура репозитория (относящаяся к развёртыванию)
- `env.example` — базовый шаблон переменных окружения; копируйте в `.env` (prod) или `.env.dev`.
- `deployment/docker/docker-compose.dev.yml` — полный локальный стек с проброшенными портами.
- `deployment/docker/docker-compose.prod.yml` — прод-стек, рассчитанный на работу за edge-прокси.
- `deployment/docker/docker-compose.npm.yml` — вариант для окружений с Nginx Proxy Manager (порты открыты).
- `deployment/nginx/dev.conf` и `deployment/nginx/prod.conf` — конфиги реверс-прокси для админки, API, сервиса дронов, MinIO, Grafana и RabbitMQ.
- `monitoring/` — конфигурации Prometheus, Grafana, Loki и Promtail.

## Настройка окружения
1. Скопируйте `env.example` в нужный файл:
   - Локальная разработка: `cp env.example .env.dev`
   - Прод: `cp env.example .env`
2. Заполните секреты и URL:
   - `JWT_ACCESS_SECRET`, `JWT_REFRESH_SECRET`, `QR_HMAC_SECRET`
   - Учётные данные `POSTGRES_*` и `DATABASE_URL` (в Compose используйте имена сервисов вроде `postgres`).
   - `MINIO_ROOT_USER`, `MINIO_ROOT_PASSWORD`, `MINIO_PUBLIC_URL`
   - `RABBITMQ_USER`, `RABBITMQ_PASSWORD`
   - `SMSAERO_EMAIL`, `SMSAERO_API_KEY`
   - `ADMIN_PANEL_URL`, `VITE_API_URL`, `VITE_MINIO_URL`, `VITE_GRAFANA_URL`
   - `DRONE_SERVICE_HTTP_URL`, `GRPC_DRONE_SERVICE_URL`, `GRPC_GO_ORCHESTRATOR_URL`
   - `FIREBASE_CREDENTIALS_FILE` — абсолютный путь внутри контейнера; Compose монтирует файл-хост по адресу `/app/secrets/firebase-service-account.json`. Если храните JSON в другом месте, поменяйте путь в Compose-файле.
3. Опционально настройте мониторинг:
   - `PROMETHEUS_PORT`, `LOKI_PORT`, `GRAFANA_PORT` и учётку Grafana.

### Секреты и внешние сервисы
- **Firebase**: расположите JSON-файл service account на хосте, скорректируйте bind mount в Compose (`go-orchestrator.volumes`), если путь отличается от `/home/takuya/googleFirebase`.
- **SMS Aero**: нужен для регистрации/логина. Если оставить пустым, приложение выведет предупреждение и отключит SMS.
- **Bootstrap админов**: переменные `ADMIN_*` и `SECOND_ADMIN_*` создают админ-аккаунты при запуске; задайте сложные пароли и валидные телефоны.

## Локальная разработка (`docker-compose.dev.yml`)
1. Убедитесь, что Docker запущен и файл `.env.dev` присутствует.
2. Запустите стек:
   ```bash
   docker compose --env-file .env.dev -f deployment/docker/docker-compose.dev.yml up --build -d
   ```
3. Проверьте основные сервисы:
   - Оркестратор (HTTP): `http://localhost:${HTTP_PORT}` (по умолчанию 8080)
   - Метрики оркестратора: `http://localhost:9091/metrics`
   - Сервис дронов (HTTP/API): `http://localhost:${DRONE_SERVICE_HTTP_PORT}` (по умолчанию 8081)
   - Метрики сервиса дронов: `http://localhost:9092/metrics`
   - Админ-панель: `http://localhost`
   - PostgreSQL: `localhost:5433`
   - Консоль MinIO: `http://localhost:9001`
   - RabbitMQ UI: `http://localhost:15672`
   - Grafana: `http://localhost:3000` (по умолчанию `admin`/`admin`, можно задать через env)
   - Prometheus: `http://localhost:9090`
   - Loki API: `http://localhost:3100`

### Состав dev-стека
| Сервис | Образ / сборка | Назначение | Примечание |
| --- | --- | --- | --- |
| `postgres` | `postgres:17-alpine` | Основная БД | Данные в томе `postgres_data_dev` |
| `minio` | `minio/minio:latest` | Хранилище S3 для QR и записей доставок | Консоль 9001, API 9000 |
| `rabbitmq` | `rabbitmq:4.2.0-rc.1-management` | Шина сообщений | Включены плагины Prometheus и management |
| `go-orchestrator` | Сборка из `backend/go-orchestrator` | REST + gRPC оркестратор | Порты 8080 и `${GO_GRPC_PORT}` |
| `drone-service` | Сборка из `backend/drone-service` | Управление дронами, WS, gRPC worker | Зависит от RabbitMQ и PostgreSQL |
| `admin-panel` | Сборка из `admin_panel` | UI операторов (Vite/React) | Отдается Nginx (порт 80) |
| `prometheus`, `loki`, `promtail`, `grafana`, `cadvisor`, `postgres-exporter`, `node-exporter` | Готовые образы стека наблюдаемости | Метрики, логи, дэшборды | Promtail читает Docker-логи |

### Полезные команды
- Логи: `docker compose -f deployment/docker/docker-compose.dev.yml logs -f go-orchestrator`
- Пересборка оркестратора: `docker compose -f deployment/docker/docker-compose.dev.yml up --build go-orchestrator -d`
- Остановка: `docker compose -f deployment/docker/docker-compose.dev.yml down`
- Удаление с томами (**внимание**: удаляет данные): `docker compose -f deployment/docker/docker-compose.dev.yml down -v`

## Продакшен (`docker-compose.prod.yml`)
1. Подготовьте хост:
   - Усильте безопасность ОС (firewall, автообновления).
   - Установите Docker и Docker Compose.
   - Настройте постоянное хранилище (volumes или bind mount с бэкапами).
2. Создайте `.env` из `env.example` и заполните прод-значениями.
3. Настройте `deployment/nginx/prod.conf`:
   - Укажите `server_name` под ваши домены.
   - Проверьте upstream-ы, если переименовывали сервисы.
   - Для TLS смонтируйте сертификаты и замените `listen 80` на `listen 443 ssl`.
4. Запуск:
   ```bash
   docker compose --env-file .env -f deployment/docker/docker-compose.prod.yml up --build -d
   ```
5. Проверка готовности:
   - `docker compose -f deployment/docker/docker-compose.prod.yml ps`
   - Откройте домен админ-панели и убедитесь, что UI отдаётся через Nginx.
   - Просмотрите логи оркестратора и сервиса дронов на наличие "HTTP server started".
   - Проверьте Prometheus (`curl http://<prometheus-host>:9090/-/ready`) и залогиньтесь в Grafana.
6. Настройте доступ к мониторингу (авторизация, доверенные домены).

### Особенности реверс-прокси
- Контейнер Nginx маршрутизирует:
  - `/api/` → оркестратор
  - `/` → админ-панель (SPA)
  - `py.*` хосты → сервис дронов (с поддержкой WebSocket upgrade)
  - Отдельные хосты для MinIO, Grafana, RabbitMQ
- Если есть внешний балансировщик/прокси:
  - Отключите встроенный Nginx.
  - Пробросьте порты оркестратора, админки, сервиса дронов, MinIO.
  - Убедитесь, что `VITE_API_URL`, `VITE_MINIO_URL`, `VITE_GRAFANA_URL` совпадают с внешними URL.

### Альтернативный прод-стек (`docker-compose.npm.yml`)
- Предназначен для инсталляций с Nginx Proxy Manager.
- Порты сервисов открыты напрямую (без внутреннего Nginx).
- Внешний прокси отвечает за TLS и маршрутизацию.
- Запуск:
  ```bash
  docker compose --env-file .env -f deployment/docker/docker-compose.npm.yml up --build -d
  ```

## Мониторинг и эксплуатация
- **Prometheus**: снимает метрики с оркестратора (`/metrics`), сервиса дронов, экспортёров; хранит в томе `prometheus_data`.
- **Grafana**: готовые дэшборды в `monitoring/grafana/dashboards`; источники данных настраиваются через provisioning.
- **Loki + Promtail**: собирают логи контейнеров; добавляйте метки через labels в Compose (`logging=promtail`).
- **cAdvisor и Node Exporter**: метрики ресурсов хоста.
- **Алертинг**: базовые правила в `monitoring/prometheus/alerts.yml`; при необходимости подключайте Alertmanager.
- **Health-check**:
  - Миграции оркестратора запускаются при старте; при ошибке контейнер падает.
  - Сервис дронов предоставляет `/health`.
  - Для PostgreSQL и MinIO настроены health-check в Compose; RabbitMQ использует `rabbitmq-diagnostics`.

## Управление данными
- **Миграции БД**: выполняются автоматически (`pkg/migrator`). При ошибке поправьте SQL и перезапустите контейнер.
- **Сиды админов**: на старте оркестратор создаёт/обновляет админов из env и генерирует QR. Очистите переменные, если сидинг не нужен.
- **Бэкапы**:
  - PostgreSQL: `docker compose exec postgres pg_dump -U $POSTGRES_USER $POSTGRES_DB > backup.sql`
  - MinIO: `mc mirror` или настройка репликации.
  - Grafana: данные в томе `grafana_data`; экспортируйте при внешнем хранении.
- **Восстановление**:
  - PostgreSQL: `psql -U $POSTGRES_USER -d $POSTGRES_DB < backup.sql`
  - MinIO: `mc mirror` обратно в бакет.

## Обновления и откаты
1. Подтяните код: `git pull origin main`.
2. Ознакомьтесь с релиз-нотами/изменениями миграций.
3. Пересоберите нужные образы:
   ```bash
   docker compose --env-file .env -f deployment/docker/docker-compose.prod.yml up --build go-orchestrator drone-service admin-panel -d
   ```
4. Следите за логами (`docker compose ... logs -f`) на предмет ошибок миграций или зависимостей.
5. Откат — checkout предыдущего коммита с повторным билдом или восстановление бэкапов.

## Устранение неполадок
- **Контейнеры перезапускаются**: смотрите логи (`docker compose ... logs <service>`). Для оркестратора ищите проблемы миграций или отсутствующие env.
- **Проблемы с JWT/аутентификацией**: проверьте секреты и синхронизацию времени (NTP).
- **Мин/МинIO редиректит на HTTPS**: `MINIO_CONSOLE_SECURE_TLS_REDIRECT` по умолчанию выключен; включайте при работе через HTTPS.
- **RabbitMQ не принимает соединения**: проверьте креды и открытые порты `5672`/`15672`.
- **Firebase push не работает**: если `FIREBASE_CREDENTIALS_FILE` пуст, отправка отключается; заполните и примонтируйте JSON.
- **CORS ошибок в админке**: обновите `ADMIN_PANEL_URL` на внешний URL.
