# Сценарий доставки от клиента до возврата дрона

Документ описывает полный цикл доставки: клиент оформляет заказ в мобильном приложении, система планирует миссию, дрон выполняет полёт, отдаёт посылку и возвращается на базовую метку `131`.

## 1. Оформление заказа (мобильное приложение)
1. Пользователь авторизуется в приложении Flutter (`POST /api/v1/auth/login`).
2. В разделе **Товары** выбирает позицию и подтверждает покупку.
3. Приложение вызывает оркестратор: `POST /api/v1/orders` с телом `{ "good_id": <UUID> }`.
4. `OrderUseCase.CreateOrder` в оркестраторе:
   - Проверяет наличие товара, снижает остаток (`good_repo`).
   - Находит подходящую ячейку постамата (`lockerRepo.FindAvailableCell`, статус → `reserved`).
   - Выбирает рабочий постамат (по приоритету/близости).
   - Пытается выделить свободный дрон (`droneRepo.GetAvailable`).
5. В БД создаётся запись заказа (`status=pending`) и доставки (привязка дрона к постамату, `status=pending`).

## 2. Планирование миссии (Оркестратор → Drone-Service)
1. Если дрон найден, оркестратор публикует `DeliveryTask` в RabbitMQ (очередь `deliveries`).
   - В payload: `delivery_id`, `order_id`, ArUco-метка, габариты, ID постамата.
2. Python `drone-service` запускает `DeliveryWorker` (`app/services/workers/delivery.py`), который читает `deliveries`.
3. `DeliveryUseCase.execute_delivery` сохраняет задачу, регистрирует дрона, вызывает WebSocket-обработчик для отправки задания.
4. Дрон подключается по WebSocket (`/ws/drone`), регистрируется по IP, получает JSON задачи.

## 3. Взлёт и перелёт дрона (ROS Clover)
1. ROS-скрипт `delivery_service.execute_delivery` инициализирует Clover API и контроллер навигации.
2. Последовательность:
   - Взлёт с текущей площадки (обычно метка `131`) до крейсерской высоты.
   - Полёт к целевой ArUco-метке (`navigate_to_marker`).
   - Переход в состояние `DELIVERING` и посадка на метку (`land_on_marker`).
3. Дрон отправляет по WebSocket статусы (`status_update`, `delivery_update`), drone-service ретранслирует их оркестратору и админ-панели.

## 4. Прибытие и сброс посылки
1. После посадки дрон отправляет `arrived_at_destination` с `order_id` и `parcel_automat_id`.
2. Drone-service вызывает gRPC `RequestCellOpen` оркестратора для открывания бронированной ячейки.
3. После подтверждения drop (команда `drop_cargo` по WebSocket):
   - Скрипт ожидает флага `cargo_drop_approved`.
   - Запускает сброс груза (симулируемый `asyncio.sleep(3)`).
   - Отправляет `cargo_dropped` с `drone_status="cargo_dropped"`.
4. Drone-service отмечает доставку как `COMPLETED`, оркестратор переводит ячейку в `available`, статус заказа → `completed`.

## 5. Возврат дрона на базу (`return_to_base`)
1. После завершения drop drone-service отправляет команду `return_to_base` с базовой меткой `131`.
2. ROS-логика взлетает (если необходимо), летит на метку `131` и садится.
3. Статус дрона → `IDLE`; оркестратор обновляет `droneRepo.UpdateStatus`.
4. Drone-service помечает задачу как выполненную, готов к следующей миссии.

## 6. Уведомления клиента
1. Оркестратор может отправить push/SMS после статуса `completed`.
2. Мобильное приложение обновляет карту заказов (polling или WebSocket).
3. Клиент видит заказ со статусом «доставлен» и данными ячейки.

## 7. Сценарий возврата (по желанию)
Если пользователь отменяет заказ (`POST /api/v1/orders/{id}/return`) при статусе `pending`/`in_progress`:
1. Оркестратор освобождает ячейку, восстанавливает остаток товара, публикует `delivery.return` (RabbitMQ).
2. `DeliveryWorker.handle_return_task` отправляет дрону команду `return_to_base` (если уже летит).
3. Дрон разворачивается и возвращается на метку `131`, заказ получает статус `cancelled`.

## Каналы обмена данными
- **REST:** мобильное приложение ↔ оркестратор (auth, заказы).
- **gRPC:** оркестратор ↔ drone-service (управление ячейками, статусы).
- **RabbitMQ:** очереди `deliveries`, `deliveries.priority`, `delivery.return` для асинхронных задач.
- **WebSocket:** дрон (ROS) ↔ drone-service (команды и телеметрия); админ-панель получает трансляции статусов.

## Точки мониторинга
- **Prometheus:**
  - `go-orchestrator` — метрики HTTP/gRPC.
  - `drone-service` — запросы FastAPI, обработка задач.
- **Loki:**
  - Логи миссий `{service="drone-service"}` и `{service="go-orchestrator"}`.
- **Grafana:**
  - Дашборды активности дронов, очередей RabbitMQ, статусов доставок.
