# Архитектура проекта

![Диаграмма архитектуры](../arch.png)

Документ описывает архитектуру платформы hiTech для доставки дронами: основные сервисы, каналы взаимодействия, хранилища данных и топологию деплоя.

## 1. Основные домены

- **Go Orchestrator (backend API):** центральная бизнес-логика — пользователи, заказы, доставки, постаматы, товары, интеграция со сторонними сервисами (SMS, push-уведомления).
- **Drone Service (Python):** прослойка между оркестратором и физическими дронами. Управляет WebSocket-сессиями, задачами доставки, телеметрией и синхронизацией состояния.
- **Парсеры оборудования:**
  - `parsers/drone` — ROS/Clover-скрипты на борту дрона (взлёт, навигация по ArUco, посадка, сброс груза).
  - `parsers/parcel_automat` — контроллер постамата (Arduino + Python), взаимодействующий с оркестратором по HTTP/gRPC.
- **Admin Panel:** веб-интерфейс (React) для операторов (мониторинг дронов, доставок, постаматов, метрик).
- **Мобильное приложение:** Flutter-клиент для пользователей/курьеров (оформление заказов, трекинг, подтверждение получения).

## 2. Сервисы и роли

### Go Orchestrator
- REST API на Gin (`/api/v1`).
- gRPC-сервер для постаматов/дронов (`pkg/pb`).
- Публикация задач в RabbitMQ (`deliveries`, `deliveries.priority`, `delivery.return`).
- Управление данными в PostgreSQL (пользователи, заказы, доставки, товары).
- Хранение файлов в MinIO (QR, записи).
- Экспорт метрик Prometheus (`/metrics`, порт 9091).

### Drone Service
- FastAPI-приложение (`/health`, `/metrics`, WebSocket для админ-панели).
- Забирает задачи из RabbitMQ, передаёт их подключённым дронам по WebSocket.
- Принимает сообщения от дрона (heartbeat, статус, сброс груза).
- Использует gRPC с оркестратором для открывания ячеек и подтверждения доставки.
- Сохраняет состояние задач в PostgreSQL, отдаёт метрики (порт 9092).

### Drone Parser (ROS)
- Запускается на Raspberry Pi/компаньон-компьютере дрона.
- Использует Clover API (`navigate`, `get_telemetry`, `land`).
- Подключается к drone-service по WebSocket, выполняет команды миссии, отчитывается о статусе.
- Реализует навигацию по ArUco и возврат на базовую метку.

### Parcel Automat
- Python-приложение, управляющее замками и QR-сканером.
- Общается с оркестратором для открытия/закрытия ячеек, проверки QR и подтверждения выдачи.
- Может быть дополнено Prometheus-экспортерами.

## 3. Каналы связи

```
Мобильное приложение ──REST──> Оркестратор <──REST── Админ-панель
                                 │
                              RabbitMQ
                                 │
                       Drone Service (FastAPI)
                                 │
                         WebSocket <──> ROS-дрон
                                 │
                         gRPC <──> Оркестратор
                                 │
                  Контроллер постамата (HTTP/gRPC)
```

- **REST:** взаимодействие клиентов и операторов с оркестратором.
- **gRPC:** оркестратор ↔ drone-service (операции с ячейками, статусы).
- **RabbitMQ:** асинхронные задачи доставки и возврата.
- **WebSocket:** real-time управление дроном и телеметрия.
- **MinIO:** общее файловое хранилище.

## 4. Хранилища и очереди

| Компонент | Назначение |
| --- | --- |
| PostgreSQL | Персистентные данные (пользователи, заказы, доставки, постаматы). |
| RabbitMQ | Очереди задач доставок, возвратов, подтверждений. |
| MinIO | S3-совместимое хранилище для QR-кодов и медиа. |
| Redis (опционально) | Может использоваться для кэша/сессий (по умолчанию отсутствует). |

## 5. Мониторинг

- **Prometheus** опрашивает оркестратор, drone-service, экспортеры RabbitMQ/Postgres, cAdvisor, node-exporter.
- **Grafana** визуализирует состояние доставок, метрики дронов, глубину очередей.
- **Loki + Promtail** собирают структурированные логи контейнеров.
- **Alertmanager** подключается через правила Prometheus.

## 6. Топология деплоя

- **Docker Compose**: `docker-compose.dev.yml`, `docker-compose.prod.yml`, `docker-compose.cicd.yml`.
- Общая сеть `skypost-delivery-network` для всех контейнеров.
- Reverse proxy Nginx маршрутизирует `/api`, `/ws`, админку, Grafana, MinIO, RabbitMQ UI.
- GitLab CI строит образы (orchestrator, drone-service, admin-panel), выкладка — `docker compose up` на целевых серверах.

## 7. Безопасность

- JWT-авторизация для клиентов API.
- TLS на уровне Nginx (обязательно в прод).
- Секреты RabbitMQ, MinIO и JWT в env-файлах.
- Firebase service account монтируется внутрь оркестратора при необходимости push-уведомлений.

## 8. Основные сценарии

1. **Создание заказа:** мобайл → оркестратор → резерв ячейки + назначение дрона → задача в RabbitMQ.
2. **Выполнение доставки:** drone-service читает задачу → WebSocket-команда → дрон выполняет миссию → статусы → оркестратор завершает заказ.
3. **Отмена/возврат:** пользователь отправляет `POST /orders/{id}/return` → оркестратор освобождает ресурсы и публикует `delivery.return` → drone-service возвращает дрона на базу → заказ `cancelled`.
4. **Мониторинг:** операторы смотрят дашборды/логи (Grafana, Loki), очередь RabbitMQ, статусы дронов в админке.

## 9. Расширяемость

- Новые сервисы можно интегрировать через RabbitMQ и gRPC.
- Дополнительные экспортеры и дашборды легко подключаются к стеку мониторинга.
- Парсер дрона расширяем для новых алгоритмов навигации или оборудования.
- Клиентские приложения используют публичные REST/gRPC API, что упрощает релизы по окружениям.
